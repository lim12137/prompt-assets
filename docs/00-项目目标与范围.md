# 项目目标与范围

## 1. 项目背景

本项目目标是建设一个可自建、可内网部署的提示词管理系统，整体体验参考用户提供的 Cherry 风格截图与现有需求描述，优先满足“简洁、易查找、易复制、可迭代更新”的日常使用场景。

当前已确认的核心诉求：

- 以分类方式浏览提示词。
- 支持顶部搜索，快速定位提示词。
- 以卡片网格形式展示提示词列表。
- 支持复制提示词内容。
- 支持查看版本历史。
- 支持点赞/评分。
- 支持用户提交更新，并进入版本历史流转。
- 支持内网 Docker 部署。

## 2. 产品目标

### 2.1 业务目标

- 为团队提供统一的提示词资产入口，降低个人散落保存和重复维护成本。
- 让非技术用户也能通过可视化界面完成查找、复制、提交改进版本等操作。
- 通过版本历史保留提示词演进过程，避免“覆盖式修改”导致内容丢失。

### 2.2 体验目标

- 首屏即可理解系统结构，不依赖培训。
- 常用操作在 3 次点击内完成：查找、查看、复制、提交更新。
- 整体视觉与操作基线贴近 Cherry 风格截图，不做后台管理系统式堆叠布局。

## 3. 界面基线

以下内容作为后续 UI 与前端实现的明确基线，不再重复讨论，除非用户后续主动调整：

- 左侧固定分类导航。
- 顶部全局搜索框。
- 主内容区采用卡片网格展示提示词。
- 页面整体为深色后台风格。
- 右上角保留核心入口：`导入`、`管理`、`创建`。

进一步拆解为实现约束：

- 左侧导航至少支持一级分类列表；是否支持二级分类，`待确认`。
- 顶部搜索默认搜索提示词标题、摘要、正文；是否支持按标签筛选，`待确认`。
- 卡片至少展示标题、分类、简述、评分/点赞、最近更新时间、复制入口。
- 深色后台优先保证对比度与长时间阅读舒适性，不以装饰性动效为重点。
- 右上角入口的职责边界：
  - `导入`：导入提示词或版本数据。
  - `管理`：进入管理视图或后台操作页。
  - `创建`：新建提示词。

## 4. MVP 范围

### 4.1 本期必须交付

- 提示词分类浏览。
- 顶部搜索。
- 卡片网格列表。
- 提示词详情查看。
- 一键复制。
- 点赞/评分展示与提交。
- 版本历史查看。
- 用户提交改进版。
- 管理员审核后发布版本。
- Docker 化内网部署方案。

### 4.2 明确不在本期

以下内容不纳入当前 MVP，避免范围失控：

- 复杂权限矩阵。
- 企业级 SSO/LDAP 深度集成，`待确认`。
- 多组织、多租户隔离，`待确认`。
- 自动化推荐、智能标签、语义检索增强。
- A/B 实验、效果评测闭环。
- 外部公开社区、开放市场、公开分享链接。
- 移动端原生 App。

## 5. 用户角色与边界

### 5.1 普通用户

- 浏览分类和卡片列表。
- 搜索提示词。
- 查看详情与版本历史。
- 复制当前版本内容。
- 提交改进版。
- 点赞/评分。

### 5.2 管理员

- 创建提示词。
- 编辑提示词元信息。
- 管理分类。
- 审核用户提交的新版本。
- 执行导入。
- 进入管理视图查看维护状态。

### 5.3 待确认角色能力

以下能力尚无明确输入，文档中只保留占位，不作为默认实现要求：

- 是否存在“审核员”独立于管理员的角色，`待确认`。
- 是否需要“只读访客”，`待确认`。
- 是否需要按部门隔离可见范围，`待确认`。

## 6. 技术与部署范围

当前已知方案方向：

- 前端与基础管理能力可参考 PromptLibrary 形态。
- 后端需补齐“一个 Prompt 对应多个 Version”的版本能力。
- 部署方式以 Docker / Docker Compose 为主。
- 使用场景为内网自建，不依赖外部 SaaS。

当前不做默认承诺的能力：

- 高可用集群与跨机房容灾，`待确认`。
- 百万级数据量性能指标，`待确认`。
- 审计合规、保密分级、审批流系统对接，`待确认`。

## 7. 成功标准

项目是否达到 MVP 目标，以以下结果判断：

- 用户可从首页直达分类、搜索、卡片列表、详情与复制操作。
- 新建提示词和提交改进版均有明确入口。
- 版本历史可追溯到提交人、时间、说明和内容版本。
- 管理员能审核并发布版本。
- 系统可在内网通过 Docker 方式启动并完成基础验收。

## 8. 待确认清单

为避免后续实现中默认补脑，以下事项需要单独确认：

- 是否支持二级分类或标签体系。
- 搜索范围是否包含版本历史全文。
- 点赞与评分是同一套机制，还是分开存储。
- 导入支持的文件格式与来源。
- 管理入口是否为独立后台页面，还是同站内切换视图。
- 登录与身份认证方式。
- 审核流程是否必须双人复核。
