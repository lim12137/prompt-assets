# 实施计划-TDD

日期：2026-04-23
阶段：项目未开始编码

## 1. 计划原则

虽然当前项目尚未开始编码，但实施仍按 TDD 思路推进：

- 先定义验收标准，再写实现
- 先写失败测试，再补最小实现
- 每完成一个垂直切片就回归测试
- 测试分层明确：单元、接口、页面流程、部署验收

这里的“测试”在当前阶段先体现为未来实施时必须落地的验收/回归任务，而不是伪造已经执行的测试结果。

## 2. 里程碑

### M1：项目骨架可运行

目标：

- 单机 Docker Compose 启动成功
- 首页、健康检查页可访问
- PostgreSQL 连接正常

### M2：Prompt 浏览链路可用

目标：

- 分类列表可展示
- Prompt 卡片列表可展示
- 详情页可查看与复制

### M3：版本链路可用

目标：

- Prompt 版本历史可查看
- 可提交新版本
- 管理员可审核通过/拒绝

### M4：互动与运营链路可用

目标：

- 点赞可用
- 排序可用
- 日志可追踪

### M5：上线验收

目标：

- 内网部署稳定
- 数据持久化
- 备份恢复可演练

## 3. 测试策略

### 3.1 单元测试

覆盖对象：

- 版本号生成
- 点赞幂等逻辑
- 审核状态流转
- 分类排序逻辑
- 搜索参数解析

要求：

- 所有领域规则必须有单元测试
- 版本状态机必须覆盖正常流和非法流

### 3.2 接口测试

覆盖对象：

- Prompt 列表接口
- Prompt 详情接口
- 版本列表接口
- 提交更新接口
- 审核通过/拒绝接口
- 点赞/取消点赞接口

要求：

- 覆盖成功、参数错误、权限不足、资源不存在四类场景

### 3.3 页面流程测试

覆盖对象：

- 首页浏览
- 搜索与分类筛选
- 详情页复制
- 投稿更新
- 管理员审核
- 点赞后列表排序变化

要求：

- 至少保留一条从普通用户到管理员审核通过的完整 E2E 流程

### 3.4 部署验收测试

覆盖对象：

- Compose 启动
- 服务健康检查
- 卷挂载持久化
- 重启后数据保留
- 备份恢复后数据一致性

## 4. 先写测试的任务拆解

## 任务 1：定义核心数据模型

先写失败测试：

- 创建 Prompt 时必须生成唯一 `slug`
- 新建 Prompt 时默认无点赞
- 初始版本号必须为 `v0001`

最小实现：

- 完成 `Prompt`、`PromptVersion`、`Category` 基本实体和持久化

回归：

- 运行单元测试，确认基础建模稳定

## 任务 2：实现 Prompt 列表与详情

先写失败测试：

- 分类筛选返回正确数据
- 搜索关键词命中标题和标签
- 详情接口返回当前发布版本

最小实现：

- 完成列表接口、详情接口、详情页展示

回归：

- 接口测试
- 页面基本浏览测试

## 任务 3：实现复制能力

先写失败测试：

- 详情页点击复制后返回成功提示
- 复制内容必须为当前发布版本正文

最小实现：

- 增加复制按钮和前端交互

回归：

- 页面流程测试

## 任务 4：实现版本历史

先写失败测试：

- 版本列表按时间倒序
- 当前发布版本标记正确
- 回滚后新版本生成而不是覆盖历史

最小实现：

- 完成版本查询、对比、回滚逻辑

回归：

- 单元测试
- 接口测试

## 任务 5：实现用户投稿更新

先写失败测试：

- 普通用户可基于当前版本提交新内容
- 提交后生成 `pending` 版本
- 未审核前不会影响当前发布版本

最小实现：

- 完成提交接口和前端表单

回归：

- 接口测试
- E2E 提交流程测试

## 任务 6：实现审核流

先写失败测试：

- 只有管理员可以审核
- 审核通过后当前发布版本更新
- 审核拒绝后状态正确保存

最小实现：

- 完成审核队列、通过/拒绝接口

回归：

- 角色权限测试
- E2E 审核流程测试

## 任务 7：实现点赞

先写失败测试：

- 同一用户重复点赞不重复计数
- 取消点赞后计数回退
- 热门排序按点赞数降序

最小实现：

- 完成点赞接口和列表排序

回归：

- 单元测试
- 接口测试
- 页面流程测试

## 任务 8：实现审计日志

先写失败测试：

- 提交更新写入日志
- 审核动作写入日志
- 回滚动作写入日志

最小实现：

- 完成日志表和关键业务埋点

回归：

- 接口测试
- 日志查询测试

## 任务 9：完成部署验收

先写失败测试：

- `docker compose up -d` 后所有核心服务健康
- 容器重启后 Prompt 数据不丢失
- 数据库备份恢复后版本链不损坏

最小实现：

- 完成 Compose、环境变量、初始化脚本、备份脚本

回归：

- 部署验收测试

## 6. 完成定义

满足以下条件才能认为一期完成：

- 核心需求全部有对应自动化测试或验收脚本
- 至少一条完整 E2E 流程稳定通过
- 单机 Compose 部署通过
- 备份恢复演练通过
- 文档补齐

## 7. 风险与应对

### 风险 1：Open WebUI 改造成本高于预期

应对：

- 在第 2 周设置止损点
- 若核心页面和审核流无法快速收敛，切换到轻量自研备选

### 风险 2：版本模型与现有底座耦合过深

应对：

- 尽量把版本逻辑独立为业务服务层
- 保持 `Prompt` 与 `PromptVersion` 表结构清晰

### 风险 3：测试滞后

应对：

- 每个任务必须先补失败测试
- 没有测试不得进入下一个里程碑

## 8. 实施建议

实施顺序建议严格按以下优先级：

1. 数据模型
2. 列表与详情
3. 版本历史
4. 投稿与审核
5. 点赞
6. 日志
7. 部署验收

原因：

- 先把浏览与版本主链打通，再叠加互动能力，风险最低。
