# 开源候选调研

核验日期：2026-04-23

目标基线：做一个可内网自建、界面尽量接近 Cherry Studio 风格的提示词管理系统，至少覆盖以下能力：

- 分类列表浏览
- 提示词详情查看与一键复制
- 版本历史
- 点赞/评分
- 用户提交更新并进入版本流转
- 单机 Docker Compose 可部署

## 评估口径

本次调研重点不是“谁的功能最多”，而是“谁最接近本项目目标，且二次开发成本最低”。因此重点看：

- 是否已有提示词管理能力，而不是泛化 AI 平台能力
- 是否支持自建与离线/内网部署
- 是否已有版本管理、权限、多人协作能力
- 是否需要引入过多与本项目无关的工作流、RAG、Agent、可观测性组件
- 是否适合作为“精简 Prompt 管理产品”的底座

## 候选一：Langfuse

仓库：

- https://github.com/langfuse/langfuse
- https://langfuse.com/docs/prompt-management/overview

已核验到的关键信息：

- Langfuse 已提供 Prompt Management，支持集中管理、版本、标签、SDK 拉取、缓存。
- 官方文档明确支持自建，且提供 Docker Compose 部署路径。
- 官方还提供 Prompt 与 GitHub 的联动能力，可触发 CI/CD 或把 Prompt 同步到仓库。
- 自建版依赖明显重于“纯提示词库”，官方文档涉及 Postgres、ClickHouse、对象存储等基础设施。

适合本项目的点：

- 版本管理能力成熟，适合严格管理生产 Prompt。
- 后续若要做评测、可观测性、Prompt 迭代数据闭环，扩展空间很大。
- 自建路径清晰，工程化程度高。

不适合本项目的点：

- 产品重心是 LLM 工程平台，不是“轻量提示词内容库”。
- 当前目标里的分类展示、点赞、用户投稿更新、类内容社区体验，并不是 Langfuse 的强项。
- 基础设施偏重，单机内网虽然能跑，但维护成本高于本项目 MVP 所需。
- 若只为了做一个精简 Prompt 管理站，Langfuse 的大量能力会闲置。

结论：

- 更适合作为“Prompt 工程平台”或二期能力底座，不适合作为当前项目的一期主选。

## 候选二：Open WebUI

仓库：

- https://github.com/open-webui/open-webui
- https://docs.openwebui.com/features/workspace/prompts/

已核验到的关键信息：

- Open WebUI 支持完全自建、离线运行。
- Workspace > Prompts 已支持 Prompt 保存、标签、访问控制、变量表单。
- 官方文档明确写明支持版本历史、回滚、共享控制。
- 项目整体同时覆盖聊天、模型接入、RAG、工具调用、管理员权限等大量通用 AI 平台能力。

适合本项目的点：

- 已有 Prompt 工作区、版本历史、权限模型，和本项目核心需求有较高重合。
- 自建门槛相对可控，容器化成熟，社区活跃度高。
- 支持 RBAC，适合内网多人共用。
- 如果接受“在现有大产品里裁剪和改造”，能较快拿到一个可用基底。

不适合本项目的点：

- Prompt 在产品中的定位更偏“聊天 Slash Command/工作区资产”，不是“独立 Prompt 库产品”。
- 现成能力里没有点赞/评分机制，也没有针对“投稿审核”的现成业务流。
- 整体产品面太大，若只是做 Prompt 管理，会带入大量无关界面和概念。
- 要做成 Cherry Studio 那种简洁提示词库，需要明显裁剪导航、交互与数据模型。

结论：

- 这是当前最接近本项目需求的现成开源底座，适合作为“主候选”。
- 但前提是接受“基于现成大项目做减法和定制”，而不是追求极简自研。

## 候选三：Dify

仓库：

- https://github.com/langgenius/dify
- https://docs.dify.ai/en/self-host/configuration/environments

已核验到的关键信息：

- Dify 的定位是开源 LLM 应用开发平台，核心是 Workflow、RAG、Agent、Prompt IDE、模型接入、监控等。
- 官方推荐 Docker Compose 启动，但默认自建栈包含 API、前端、Nginx、数据库、Sandbox、SSRF Proxy、插件守护进程等多个服务。
- Prompt 能力存在，但它是应用编排能力的一部分，不是以“轻量 Prompt 资产库”为中心。

适合本项目的点：

- 若后续目标是做“Prompt + Workflow + 知识库 + Agent”一体化平台，Dify 很强。
- 自建方案完整，生态成熟，团队协作与运维能力较强。

不适合本项目的点：

- 对当前项目明显过重，系统边界远大于“提示词管理”。
- Prompt 资产管理不是产品主线，分类库、点赞、版本投稿等内容型体验需要较多自定义。
- 单机 Compose 虽可运行，但组件数量、配置复杂度、升级成本都高。
- 引入 Dify 很容易把项目带偏成“内部 AI 应用平台”，而不是“简洁 Prompt 管理器”。

结论：

- 不建议作为当前项目的一期方案。
- 仅在项目目标扩大到“工作流/Agent 平台”时再重新评估。

## 候选四：yinpu/prompt_manager

仓库：

- https://github.com/yinpu/prompt_manager

已核验到的关键信息：

- 这是一个 Python 提示词版本管理库，核心模型是 `Project > Prompt > Version`。
- 仓库展示了基于文件系统的目录结构：每个版本保存 `prompt.txt`、`outputs.json`、`meta.json`。
- 支持自动版本号、自定义版本号、导入导出、多模型输出记录。
- 当前更像一个库/原型项目，而不是可直接交付给业务用户的完整 Web 产品。

适合本项目的点：

- 版本模型非常贴合本项目，尤其适合作为后端版本引擎的设计参考。
- 基于文件系统的版本快照天然适合内网、低依赖、易备份场景。
- 代码量和概念都轻，适合做极简自研底座。

不适合本项目的点：

- 没有现成前端、权限、审核流、点赞、分类首页等产品能力。
- 需要自行补齐 Web API、管理后台、用户界面、认证授权。
- 仓库成熟度和社区体量明显弱于 Open WebUI、Dify、Langfuse。

结论：

- 不适合作为“拿来即用”的开源产品。
- 非常适合作为轻量自研方案里的“版本模型参考”或“后端实现思路”。

## 对比结论

| 候选 | 贴合当前目标 | 自建友好 | 版本能力 | UI/业务贴合度 | 系统复杂度 | 结论 |
| --- | --- | --- | --- | --- | --- | --- |
| Langfuse | 中 | 中 | 高 | 低 | 高 | 更适合 LLM 工程平台，不适合一期主选 |
| Open WebUI | 高 | 高 | 高 | 中 | 中高 | 当前最适合作为主候选 |
| Dify | 低 | 中 | 中 | 低 | 很高 | 当前不建议采用 |
| yinpu/prompt_manager | 中 | 高 | 中高 | 很低 | 低 | 适合作为轻量自研备选思路 |

## 本文结论

- 如果优先要“最快落地一个可用系统”，应优先看 `Open WebUI 二次开发`。
- 如果优先要“极简、可控、贴业务、长期维护成本低”，应参考 `yinpu/prompt_manager` 的版本模型做轻量自研。
- `Langfuse` 和 `Dify` 都不是不能做，而是都明显超出当前项目边界；前者偏工程平台，后者偏应用平台。
