# 候选筛选与结论

核验日期：2026-04-23

## 项目判断前提

当前项目不是要做一个“通用 AI 平台”，而是要做一个：

- 面向内网用户
- 风格简洁
- 以 Prompt 资产管理为中心
- 可追踪版本
- 支持投稿更新
- 支持点赞/评分
- 单机 Docker Compose 就能上线

因此筛选标准按优先级排序如下：

1. 是否围绕 Prompt 资产管理，而不是泛化 AI 平台。
2. 是否能快速收敛到简洁界面，而不是需要大幅裁剪复杂产品。
3. 是否支持内网单机自建。
4. 是否具备版本能力或容易补齐版本能力。
5. 二次开发成本和后续维护成本是否可控。

## 候选筛选结果

### 1. Open WebUI

判断：

- 保留进入决赛。

原因：

- 已有 Prompt Workspace、版本历史、回滚、共享控制、标签、权限。
- 自建和离线部署成熟，内网适配较好。
- 社区活跃，后续碰到问题更容易找到资料。
- 问题在于产品过大，不是为“Prompt 内容库”单点设计，需要做明显裁剪。

### 2. Langfuse

判断：

- 不作为一期主选，但保留为二期工程化方向参考。

原因：

- Prompt 管理、版本、GitHub 联动、评测与可观测性都很强。
- 但当前项目需要的是“轻量内容管理体验”，不是 LLM 工程平台。
- 基础设施和运维心智负担偏重，不利于最小可用版本快速落地。

### 3. Dify

判断：

- 本轮淘汰。

原因：

- Dify 很强，但它强在工作流、Agent、RAG、应用编排。
- 本项目当前边界过窄，采用 Dify 会引入过多无关服务和概念。
- 用它做 Prompt 管理器，本质上是在拿大型平台反向裁成小工具，不经济。

### 4. yinpu/prompt_manager

判断：

- 作为轻量自研备选保留。

原因：

- `Project > Prompt > Version` 模型很适合本项目。
- 文件系统快照非常适合内网、备份、审计、离线迁移。
- 但它不是完整产品，只能作为“版本层参考”或“后端底座思路”。

## 最终建议

### 主建议：基于 Open WebUI 二次开发

建议级别：主选

适用前提：

- 目标是 4 到 8 周内拿到一个可演示、可上线试运行的版本。
- 接受“保留其用户/权限/版本能力，替换或裁剪部分 UI 与业务流”。

推荐原因：

- 现成能力与本项目目标重合度最高。
- 版本、权限、多人使用、自建部署这些难点已经有基础。
- 一期只需新增或改造以下差异能力：
  - 分类首页和卡片式浏览
  - 一键复制体验优化
  - 点赞/评分
  - 投稿更新与审核流
  - 面向 Prompt 库的导航裁剪

不选 Langfuse 作为主选的原因：

- Langfuse 强项在 Prompt 工程化，而不是 Prompt 内容库体验。

不选 Dify 作为主选的原因：

- Dify 系统范围远大于当前需求。

不选 yinpu/prompt_manager 作为主选的原因：

- 从零补 UI、认证、权限、审核流，一期交付速度不如直接改造 Open WebUI。

### 备建议：基于 yinpu/prompt_manager 思路做轻量自研

建议级别：备选

适用前提：

- 目标是做一个长期极简、完全贴合业务、不想背大型开源项目包袱的系统。
- 可以接受一期开发更多基础能力。

推荐原因：

- 最容易做成“真正简洁”的 Prompt 管理产品。
- 文件系统版本快照天然适合备份、审计、回滚。
- 内网单机 Compose 很容易控制依赖，长期维护简单。

不作为主选的原因：

- 一期开发量更大，尤其是权限、前端管理界面、审核流、操作日志都要自己补。

## 决策建议

推荐采用如下决策：

- 一期：`Open WebUI 二次开发`
- 二期保底退路：如果改造中发现 Open WebUI 裁剪成本过高，切换到 `轻量自研 + yinpu/prompt_manager 版本模型`

切换阈值建议：

- 若第 2 周结束前仍无法低成本完成“分类列表 + Prompt 详情 + 点赞 + 投稿审核”核心交互，则立即止损，转备选方案。

## 交付判断

若本周就要启动实施，建议直接按主建议推进；原因不是它最完美，而是它最可能在当前阶段以可控风险交付出第一个可运行版本。
