# 2026-04-24 二期创建/导入/迭代聚合设计

## 1. 文档目标

在一期“浏览-投稿-审核”闭环基础上，定义二期围绕创建、导入、候选迭代、详情聚合展示的统一设计基线，确保产品、后端、前端、测试对同一语义达成一致。

本设计只定义方案，不包含代码实现。

## 2. 已确认基线（本设计约束）

1. 管理员后台创建新 Prompt 首版。  
2. 管理员后台 JSON 批量导入新 Prompt 首版。  
3. 员工对已有 Prompt 提交候选迭代，候选号形如“基准正式版 + cand + 用户名 + 次数”。  
4. 每个提示词详情聚合展示 1 张官方推荐卡 + 各员工各自最新 1 张候选卡。  
5. 页面展示标题统一中文，slug 仅内部使用。  
6. 候选卡支持一键复制。

## 3. 范围与边界

## 3.1 二期范围（In Scope）

- 管理后台：创建单个 Prompt、批量 JSON 导入 Prompt 首版。
- 员工迭代：提交候选版本并按新规则生成候选号。
- 详情页：官方推荐卡 + 员工候选聚合卡展示。
- 复制交互：候选卡一键复制正文。

## 3.2 非范围（Out of Scope）

- 多级审批流、会签流。
- 公网认证体系重构（沿用现有最小角色语义）。
- 多租户隔离与跨组织数据共享。
- 复杂导入模板编排器（仅支持定义好的 JSON 结构）。

## 4. 角色与权限

- 管理员（admin）
  - 可创建新 Prompt 首版。
  - 可执行 JSON 批量导入。
  - 可继续保留审核能力（一期已有）。
- 员工（user）
  - 仅可对已有 Prompt 提交候选迭代。
  - 不可直接创建新 Prompt 主体。

## 5. 领域模型与命名规则

## 5.1 核心实体语义

- Prompt：提示词资产主体，页面展示使用中文标题。
- 官方推荐版（official）：当前被推荐的正式版本（等价于现有“当前正式版”语义）。
- 员工候选版（candidate）：员工提交的迭代版本，不直接覆盖官方推荐版。

## 5.2 候选号规则（强约束）

候选号格式：

`{官方基准版号}-cand-{用户名}-{次数}`

示例：

- `v0003-cand-zhangsan-1`
- `v0003-cand-zhangsan-2`
- `v0003-cand-lisi-1`

规则说明：

1. `官方基准版号`：提交时该 Prompt 的官方推荐版号（如 `v0003`）。
2. `用户名`：标准化用户标识（建议小写、仅字母数字与下划线）。
3. `次数`：同一用户、同一基准版下自增计数（从 1 开始）。
4. 若官方推荐版升级（如 `v0004`），候选计数重新按新基准版独立计算。

## 5.3 slug 与中文展示规则

- slug 仅用于内部唯一标识、路由与接口参数，不在主 UI 显式作为标题展示。
- 页面主展示统一使用中文标题字段（title）。
- 导入与创建时都必须提供中文标题；slug 可自动生成或显式传入（以系统校验结果为准）。

## 6. 数据设计（在现有模型上的二期扩展）

## 6.1 复用现有表

- `prompts`：继续作为主体表。
- `prompt_versions`：继续承载版本正文与来源类型。
- `submissions`：继续承载审核相关关系（如后续仍需审核）。

## 6.2 二期建议新增字段（设计建议）

在 `prompt_versions` 增加：

- `is_official_recommended`（boolean，默认 false）
  - 标记是否为当前官方推荐候选来源中的官方卡（也可通过 `prompts.current_version_id`推导，字段可选）。
- `candidate_owner`（text，可空）
  - 员工候选归属用户名（官方版为空）。
- `candidate_base_version_no`（text，可空）
  - 候选提交时参考的官方基准版号。
- `candidate_seq`（int，可空）
  - 候选次数序号。
- `candidate_label`（text，可空，唯一）
  - 存储完整候选号，如 `v0003-cand-zhangsan-2`。

约束建议：

- 当 `source_type='submission'` 时，`candidate_owner/candidate_base_version_no/candidate_seq/candidate_label` 必填。
- 唯一约束：`(prompt_id, candidate_label)`。
- 索引建议：`(prompt_id, candidate_owner, submitted_at desc)` 用于“每员工最新候选”聚合。

说明：若不新增字段，也可把候选号塞入 `version_no`，但会污染官方版号体系，不建议。

## 7. 功能设计

## 7.1 管理员后台创建新 Prompt 首版

### 输入字段（最小）

- 中文标题（必填）
- slug（可选，空则自动生成）
- 摘要（必填）
- 分类（必填）
- 首版正文（必填）

### 行为

1. 创建 Prompt 主体记录。
2. 创建首版 `v0001`，`source_type='create'`。
3. 该首版作为官方推荐卡来源。
4. 写入审计日志 `prompt.created`。

### 输出

- 创建成功后跳转到该 Prompt 详情页，默认看到“官方推荐卡”。

## 7.2 管理员后台 JSON 批量导入新 Prompt 首版

### JSON 结构（建议）

```json
{
  "items": [
    {
      "title": "中文标题",
      "slug": "optional-slug",
      "summary": "摘要",
      "categorySlug": "programming",
      "content": "首版正文"
    }
  ]
}
```

### 导入规则

1. 逐条校验字段完整性。
2. slug 冲突策略：默认跳过并记录失败原因（不覆盖）。
3. 成功项创建 Prompt + `v0001(create)`。
4. 返回导入报告（成功数、失败数、失败明细）。

### 失败处理

- 不要求全有或全无事务；采用“逐条结果可追踪”的批处理模式。

## 7.3 员工提交候选迭代

### 前置条件

- 目标 Prompt 必须存在。
- 当前必须存在官方推荐版（基准版）。

### 行为

1. 读取当前官方基准版号（如 `v0003`）。
2. 计算该用户在该基准版下的下一个候选次数。
3. 生成候选号（`v0003-cand-zhangsan-2`）。
4. 创建候选版本（`source_type='submission'`）并写入候选元信息。

### 输出

- 返回候选号、基准版号、候选归属人。

## 7.4 详情聚合展示

详情页卡片区展示：

1. 官方推荐卡（固定 1 张）
  - 展示：中文标题、官方版号、摘要、正文、复制按钮。
2. 员工候选卡（每员工最新 1 张）
  - 聚合规则：按 `candidate_owner` 分组，每组取 `submitted_at` 最新。
  - 展示：候选号、员工名、提交时间、正文、复制按钮。

排序建议：

- 官方推荐卡置顶。
- 候选卡按“提交时间倒序”排列。

## 7.5 候选卡一键复制

- 每张候选卡提供“复制候选版本”按钮。
- 复制对象为该候选卡正文全文。
- 交互反馈：
  - 成功：`已复制候选版本内容`
  - 失败：`复制失败，请手动复制`

## 8. API 设计草案（二期）

## 8.1 管理端创建

- `POST /api/admin/prompts`
  - 入参：`title, slug?, summary, categorySlug, content`
  - 出参：`promptSlug, officialVersionNo`

## 8.2 管理端批量导入

- `POST /api/admin/prompts/import`
  - 入参：`{ items: [...] }`
  - 出参：`{ total, success, failed, errors[] }`

## 8.3 员工候选提交（升级现有提交接口）

- `POST /api/prompts/{slug}/submissions`
  - 出参增加：`candidateLabel, candidateBaseVersionNo, candidateOwner`

## 8.4 详情聚合视图

- `GET /api/prompts/{slug}`
  - 出参增加：
    - `officialCard`
    - `latestCandidateCardsByUser[]`

## 9. 前端页面设计草案

## 9.1 管理后台

- 新增“创建 Prompt”入口（表单）。
- 新增“批量导入”入口（JSON 文本框 + 校验提示 + 导入报告）。

## 10. 测试设计（TDD 建议）

## 10.1 Integration

1. `POST /api/admin/prompts` 成功创建 `v0001(create)`。  
2. 创建时 slug 冲突返回 409。  
3. `POST /api/admin/prompts/import` 混合成功/失败导入报告正确。  
4. 员工提交候选时候选号规则正确（同用户同基准版计数自增）。  
5. 详情聚合返回“1 张官方卡 + 每员工最新 1 张候选卡”。

## 10.2 E2E

1. 管理员从后台创建 Prompt，详情可见官方推荐卡。  
2. 管理员 JSON 批量导入后列表增加并可打开详情。  
3. 员工多次提交后，详情只展示其最新候选卡。  
4. 候选卡复制按钮成功反馈可见。

## 10.3 数据回归

- 真实 DB 下验证候选号唯一性与聚合查询性能（至少千级候选记录下可接受）。

## 11. 风险与缓解

1. 候选号并发冲突  
- 风险：同用户并发提交可能产生相同次数。  
- 缓解：数据库唯一约束 + 重试生成策略。

2. 导入质量不稳定  
- 风险：JSON 字段缺失、slug 非法、分类不存在。  
- 缓解：导入前校验 + 逐条错误报告。

3. 详情聚合查询变慢  
- 风险：候选数量增长导致分组查询成本上升。  
- 缓解：建立分组索引，必要时增加物化视图或缓存层。

## 12. 交付顺序建议

1. 后端先行：创建 API + 导入 API + 候选号规则 + 聚合查询。  
2. 前端接入：管理后台创建/导入界面 + 详情聚合卡。  
3. 测试收口：integration + e2e + 真实 DB 回归，报告落盘到 `docs/*.md`。
