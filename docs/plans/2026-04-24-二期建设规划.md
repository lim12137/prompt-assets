- 一期状态：MVP 主链路（浏览、详情、投稿、审核、Compose 部署）已完成并验收收口，见 `docs/验收记录-20260424.md`。

---

## 1）当前产品/技术缺口盘点

### 1.1 产品能力缺口（面向二期）

1. 管理端能力仍偏最小闭环  
- 已有：待审核列表、approve/reject。  
- 缺口：分类管理、提示词条目管理、审核检索/筛选、批量处理、操作回溯入口。

2. 导入/创建入口仍为占位反馈  
- 首页 `导入`、`创建` 当前仅提示“暂未实现”，对运营上线价值有限。

3. 权限模型过于简化  
- 当前核心是请求头 `x-user-role` 判定 admin，适合内测，不适合作为稳定生产边界。  
- 缺少登录鉴权、会话管理、最小权限矩阵与审计联动。

4. 审核与版本治理缺少“流程治理”能力  
- 缺少审核意见模板、驳回理由结构化、回滚入口、状态筛选、历史对比视图。

### 1.2 技术与工程缺口

1. 测试分层存在“真实链路覆盖不足”  
- 虽已新增 admin 真实 DB e2e 脚本，但尚未形成默认回归门禁。  
- 仍有大量 integration 依赖 fixture（为开发效率服务），与生产数据行为存在偏差窗口。

2. 数据初始化标准化不足  
- 已有 `db:test:prepare` 能完成测试库 up/migrate/seed，但生产/预发与本地的初始化流程还未统一成单一脚本契约。

3. CI/CD 缺口明显  
- 目前仅 `publish-images.yml`，没有 PR 阶段自动执行：
  - 单元/集成/API 测试
  - 关键 e2e（含 admin 链路）
  - 构建与迁移校验

4. 可观测性与运维基线不足  
- 缺少结构化业务指标（审核队列长度、审核耗时、失败率）。  
- 缺少最小告警、备份恢复演练与日志查询指引。

## 2）二期目标与边界

## 2.1 二期目标（Phase 2）

1. 从“可用 MVP”提升到“可持续运营”  
- 管理后台从单点审核闭环扩展为可运维的内容治理面板。

2. 从“可跑”提升到“可验证、可发布”  
- 建立真实 DB 驱动的稳定测试门禁并接入 CI，减少环境漂移。

3. 从“内测权限”提升到“基础生产权限”  
- 交付最小鉴权体系与角色权限矩阵，保障管理动作可控、可追责。

4. 保持单体架构、控制复杂度  
- 二期仍维持单仓 + Next.js + PostgreSQL + Compose，不做微服务拆分。

## 2.2 二期边界（不做）

- 不做多租户与组织级隔离。
- 不做复杂工作流引擎（多级会签、跨团队审批编排）。
- 不做大规模云原生改造（K8s、服务网格、多活容灾）。
- 不做“泛 AI 平台”扩张（聊天编排、模型路由中台等）。

## 3）架构策略与关键决策（ADR 摘要）

### ADR-P2-01：保持单体架构，先补治理与质量门禁

- 决策：二期不拆服务，继续在现有单体内完成管理治理与测试体系升级。  
- 理由：当前主要风险不是吞吐瓶颈，而是流程稳定性与发布质量。  
- 代价：模块边界治理要靠代码组织与测试约束，而非服务边界天然隔离。

### ADR-P2-02：测试以“真实 DB 回归”为发布门禁，fixture 作为开发加速层

- 决策：保留 fixture 测试用于快速反馈，同时新增真实 DB 套件作为 CI 阻断门槛。  
- 理由：兼顾速度与真实性，避免仅 fixture 导致行为偏差漏检。  
- 代价：CI 时长上升，需要分层执行策略（PR 快速集、主干全量集）。

### ADR-P2-03：权限体系最小可用化

- 决策：引入最小认证与会话，不再仅依赖明文 header 角色判定。  
- 理由：管理动作已进入真实运营场景，需要基本安全边界。  
- 代价：需补登录态、角色中间件、权限回归测试。

## 4）可执行任务拆解（优先级/风险/验证）

## 4.1 P0（必须先做，二期门槛）

### Task P2-01：CI 增加测试门禁工作流

- 优先级：P0  
- 范围：
  - 新建 `ci-verify` 工作流（PR + push 到主干）。
  - 至少执行：`build:web`、关键 integration、`test:e2e:admin:db`（或等价真实 DB admin 套件）。  
- 风险：
  - Docker in CI 稳定性、时长增加。
  - 端口/环境变量漂移导致偶发失败。  
- 验证方式：
  - 新 PR 必须看到 CI 绿色才能合并。
  - 故意引入 admin 审核链路回归，CI 能阻断。

### Task P2-02：统一 DB 初始化契约（dev/test/ci）

- 优先级：P0  
- 范围：
  - 固化 `up -> migrate -> seed -> verify` 一键脚本约定。
  - 将数据库准备步骤标准化到文档与脚本入口。  
- 风险：
  - 多套库名/端口配置冲突（`prompt_assets` vs `prompt_management_test`）。  
- 验证方式：
  - 本地与 CI 使用同一准备命令，结果一致。
  - seed 结果可验证（pending submissions、prompt 数量等）。

### Task P2-03：权限与鉴权最小闭环

- 优先级：P0  
- 范围：
  - 定义角色矩阵（viewer/editor/admin）。
  - 接入最小登录态/会话校验（至少管理端与 admin API）。  
- 风险：
  - 影响现有测试与接口调用方式，回归面较大。  
- 验证方式：
  - 非 admin 调用管理 API 返回标准拒绝码。
  - e2e 验证 admin 可达、普通用户不可达。

## 4.2 P1（二期核心能力）

### Task P2-04：管理后台能力扩展

- 优先级：P1  
- 范围：
  - 审核列表筛选（pending/approved/rejected）、检索、分页。
  - 审核详情页（候选版本对比、审核意见展示）。  
- 风险：
  - 状态复杂度增加，前后端契约需稳定。  
- 验证方式：
  - integration 覆盖筛选/状态流转。
  - e2e 覆盖“检索 -> 审核 -> 列表刷新”。

### Task P2-05：导入/创建功能落地（替代占位按钮）

- 优先级：P1  
- 范围：
  - 导入最小版本（CSV/JSON 二选一）。
  - 创建新提示词条目并进入待审核或直接草稿。  
- 风险：
  - 导入数据质量与幂等问题。  
- 验证方式：
  - 端到端导入成功率、重复导入行为断言。
  - 审核链路与审计日志联动验证。

### Task P2-06：审计与可观测性增强

- 优先级：P1  
- 范围：
  - 审核行为指标（通过率、平均处理时长、积压量）。
  - 错误日志与关键告警（管理 API 失败率阈值）。  
- 风险：
  - 指标定义不一致导致数据不可用。  
- 验证方式：
  - 指标口径文档 + 查询脚本 + 演示面板。

## 4.3 P2（优化项）

### Task P2-07：发布与回滚操作手册化

- 优先级：P2  
- 范围：
  - 镜像版本发布流程、DB 变更回滚流程、故障恢复 SOP。  
- 风险：
  - 文档与实际流程脱节。  
- 验证方式：
  - 演练一次“故障回滚”，并落盘报告。

### Task P2-08：并发与性能基线重建

- 优先级：P2  
- 范围：
  - 固化 admin 与投稿场景的 k6 基线阈值。  
- 风险：
  - 环境差异导致阈值不稳定。  
- 验证方式：
  - 同一环境连续三次压测结果波动在可接受区间。

## 6）验收口径（Phase 2 Done Definition）

- 测试门禁：
  - `build:web` 通过；
  - 关键 integration 通过；
  - `test:e2e:admin:db` 在 CI 稳定通过。  
- 安全与权限：
  - 管理端与 admin API 均受鉴权保护；
  - 权限矩阵有自动化回归用例。  
- 产品能力：
  - 管理页支持筛选/检索/审核详情；
  - 导入与创建完成最小可用闭环。  
- 运维能力：
  - 发布、回滚、恢复流程有可执行文档并演练记录。

---

## 7）核心建议（执行顺序）

1. 先做“质量门禁”再做“功能扩展”  
- 如果 CI 与真实 DB 回归不先稳住，二期功能越多，回归成本越高且风险不可控。

2. 权限体系与管理后台扩展并行推进，但以权限先落地为前置条件  
- 管理功能扩展必须建立在可控权限边界上，避免先扩权后补安全。

3. 导入/创建属于业务增量，不应抢占 P0  
- 建议在 M2 后推进，避免影响二期稳定性主线。
