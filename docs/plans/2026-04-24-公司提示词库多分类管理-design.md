# 2026-04-24 公司提示词库多分类管理设计

## 1. 文档目标

基于当前二期已落地的“创建/导入/迭代/聚合”能力，定义下一步“公司提示词库”在分类管理与前台筛选上的统一产品与技术方案，确保产品、后端、前端、测试在同一语义下推进。

本设计只定义方案，不包含代码实现。

## 2. 已确认基线（本设计约束）

1. 页面名称改为“公司提示词库”。
2. 分类改为管理端可增删。
3. 提示词支持多选分类标签（一个提示词可挂多个分类）。
4. 删除分类前需二次确认，并显示受影响提示词数量。
5. 删除分类后，零分类提示词自动归入系统“待分类”。
6. “待分类”默认折叠，且不能手动勾选。
7. 前台多选分类筛选采用 OR 逻辑。

## 3. 范围与边界

## 3.1 范围（In Scope）

- 首页与管理页文案统一切换为“公司提示词库”语义。
- 分类主数据从“固定枚举”升级为“可管理实体”。
- 提示词与分类关系从单选改为多选标签关系。
- 分类删除影响评估、确认交互与删除后兜底归档规则。
- 首页分类筛选从单选改为多选 OR 逻辑。
- “待分类”分类的系统保护规则（自动归入、默认折叠、不可手动勾选）。

## 3.2 非范围（Out of Scope）

- 分类层级（父子分类）与分类树。
- 标签权重、排序推荐、个性化分类推荐。
- 跨租户分类隔离（仍沿用当前单租户语义）。
- 分类国际化多语言翻译能力。

## 4. 产品规则定义

## 4.1 命名与展示

- 首页主标题统一为“公司提示词库”。
- 管理端涉及资产列表、创建导入入口等标题与导航文案统一为“公司提示词库”语义。
- slug 继续仅内部使用，不作为主标题展示。

## 4.2 分类生命周期

- 管理员可新增分类：至少包含 `name`（中文显示名）与内部 `slug`（可自动生成）。
- 管理员可删除分类：删除前必须显示“将影响 N 条提示词”并要求确认。
- 系统保留分类“待分类”（系统内置，以下简称 `uncategorized`）：
  - 不允许删除。
  - 不允许在提示词编辑/创建表单中被手动选择。
  - 仅作为“兜底自动归类”目标。

## 4.3 提示词多分类规则

- 每个提示词至少属于 1 个分类（业务不允许长期零分类）。
- 常规情况下，用户可手动选择 1..N 个“普通分类”。
- 若某提示词因分类删除导致失去全部分类，系统自动补挂 `uncategorized`。

## 4.4 前台筛选规则

- 分类筛选支持多选。
- 命中逻辑为 OR：提示词只要命中任一已选分类即可展示。
- 未选择任何分类时，展示全部提示词（再叠加关键词等现有筛选条件）。
- “待分类”在筛选面板中默认折叠显示，不提供手动勾选入口。

## 5. 数据模型设计

## 5.1 分类实体（categories）

建议字段：

- `id` (pk)
- `slug` (unique, 内部标识)
- `name` (中文展示名)
- `is_system` (boolean, 标识系统分类)
- `is_selectable` (boolean, 标识是否可在 UI 手动选择；`uncategorized=false`)
- `is_collapsed_by_default` (boolean, 仅前台展示提示，`uncategorized=true`)
- `created_at`, `updated_at`

约束建议：

- `uncategorized` 仅允许存在 1 条（可用唯一约束 + seed 保证）。
- `is_system=true` 的分类默认不可删除。

## 5.2 提示词-分类关系（prompt_categories）

新增关系表（多对多）：

- `prompt_id` (fk -> prompts.id)
- `category_id` (fk -> categories.id)
- `created_at`

约束建议：

- 唯一约束：`(prompt_id, category_id)`，防止重复打标。
- 索引：`category_id`、`prompt_id`。

## 5.3 迁移策略（从单分类到多分类）

迁移步骤建议：

1. 保留现有 `prompts.category_id` 作为迁移输入。
2. 为每条历史 prompt 写入一条 `prompt_categories` 关系。
3. 读路径先支持新旧双读，校验后切换到关系表为准。
4. 稳定后再考虑移除 `prompts.category_id`（可后置，不强制本阶段完成）。

## 6. 接口设计草案

## 6.1 分类管理接口（管理员）

- `GET /api/admin/categories`
  - 返回分类列表与统计信息（可含每类关联提示词数）。
- `POST /api/admin/categories`
  - 入参：`name`，可选 `slug`
  - 出参：新分类对象
- `DELETE /api/admin/categories/{slug}`
  - Query 或 body 可带 `confirm=true`
  - 返回：删除分类信息 + 受影响提示词数 + 自动归入待分类数量
  - 对系统分类返回 403（如 `uncategorized`）

## 6.2 提示词创建/更新接口扩展

- 创建与导入入参支持 `categorySlugs: string[]`（替代单一 `categorySlug`）。
- 为兼容历史调用，短期可保留单值字段并在服务端转换为数组。
- 校验规则：
  - 数组不能为空。
  - 不能包含 `uncategorized`（手动提交场景）。
  - 全部分类必须存在。

## 6.3 前台列表接口扩展

- `GET /api/prompts?categories=a,b,c`
  - `categories` 为多值筛选参数。
  - 语义：OR 命中（`IN` 关系）。
  - 与关键词搜索组合时使用 AND（关键词条件 ∧ 分类 OR 条件）。

## 7. 前端交互设计

## 7.1 首页（公司提示词库）

- 标题改为“公司提示词库”。
- 分类区域改为多选组件。
- “待分类”项默认折叠在“系统分类”区域，不显示可勾选 checkbox。
- 当前筛选条件可视化展示已选普通分类标签。

## 7.2 管理端分类管理

- 新增“分类管理”区块（可在 `/admin` 或独立页面）：
  - 新增分类表单。
  - 分类列表（名称、slug、关联提示词数、系统标识）。
  - 删除按钮（系统分类禁用）。
- 删除流程：
  1. 点击删除。
  2. 拉取/显示影响数（例如“将影响 27 条提示词”）。
  3. 二次确认后执行删除。
  4. 反馈结果（删除成功 + 自动归入待分类数量）。

## 7.3 创建/导入表单

- 分类输入升级为“多选标签”。
- `uncategorized` 不在可选列表中出现。
- 导入 JSON 支持多分类字段（建议 `categorySlugs`）。

## 8. 关键流程

## 8.1 删除分类流程（含兜底）

1. 管理员发起删除分类 `C`。
2. 系统校验 `C` 不是系统分类。
3. 计算影响提示词集合 `P`（当前挂 `C` 的提示词）。
4. 对每个 `p in P`：
  - 删除 `(p, C)` 关系。
  - 若 `p` 删除后无任何分类，则补挂 `(p, uncategorized)`。
5. 返回删除统计结果并写审计日志。

## 8.2 首页筛选流程（OR）

1. 用户勾选分类集合 `S={c1,c2,...}`。
2. 查询条件构造为：`prompt has any category in S`。
3. 若 `S` 为空，则不附加分类过滤。

## 9. 测试策略（验收导向）

## 9.1 Integration

1. 分类新增成功；重名或 slug 冲突返回 409。
2. 删除普通分类前返回影响数，确认后删除成功。
3. 删除后零分类提示词自动归入 `uncategorized`。
4. 删除系统分类 `uncategorized` 返回 403。
5. 创建/导入支持多分类；提交 `uncategorized` 被拒绝。
6. 列表接口多分类筛选为 OR。

## 9.2 E2E

1. 首页标题显示“公司提示词库”。
2. 首页可多选分类筛选，命中 OR 逻辑。
3. `uncategorized` 默认折叠且无可勾选入口。
4. 管理端新增分类成功并可在创建页被选择。
5. 管理端删除分类弹确认并显示影响数，执行后受影响提示词进入待分类。

## 9.3 回归重点

- 现有审核流、候选迭代、详情聚合不受分类多选改造影响。
- 导入与创建的旧调用（单分类）在兼容期内仍可工作。

## 10. 风险与缓解

1. 数据一致性风险（删除分类并发操作）
- 缓解：删除流程放入事务；关系删除与待分类补挂原子化。

2. 查询性能风险（多分类 OR）
- 缓解：关系表索引 + 去重查询；必要时增加分页与缓存。

3. 迁移期双写/双读复杂度
- 缓解：明确阶段开关；先双读校验再收敛单读。

4. 系统分类被误操作
- 缓解：后端强校验 `is_system`，前端仅做辅助禁用。

## 11. 交付顺序建议

1. 数据层：关系表与 `uncategorized` 系统分类基线。
2. 后端：分类 CRUD、删除兜底、列表 OR 筛选、创建导入多分类入参。
3. 前端：标题与筛选 UI、管理端分类管理、创建导入多选分类。
4. 测试：integration + e2e + 真实 DB 回归并落盘报告。

## 12. 完成判定（DoD）

- 首页与管理核心文案完成“公司提示词库”切换。
- 分类可在管理端新增/删除，删除前有影响确认。
- 删除分类后无分类提示词自动归入系统“待分类”。
- `uncategorized` 默认折叠且不能手动勾选。
- 提示词支持多分类标签，前台多分类筛选为 OR。
- 自动化测试覆盖上述关键规则并通过。
