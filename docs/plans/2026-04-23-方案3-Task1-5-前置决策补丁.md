# 方案3 Task 1~5 前置决策补丁

文档版本：v1.0  
日期：2026-04-23  
适用范围：方案3（极简提示词资产平台）Task 1~5

## 背景

Task 1~5 进入实现前，当前文档在角色枚举、审核对象语义、测试数据库运行方式、`packages/db` 最小依赖与脚本约定上存在口径差异。  
本补丁用于一次性统一执行口径，避免实现与测试返工。

## 决策项

## DCP-001 `users.role` 固定枚举

- 冻结为：`user | admin`。
- 不引入 `viewer/editor` 或其他角色别名。
- 鉴权、测试数据、接口权限断言均以该二值枚举为唯一来源。

## DCP-002 Task 5 一次性引入 `submissions`

- Task 5 必须完整引入 `submissions`（表、关系、API、测试）。
- 语义冻结：
- `submission` 是审核对象（队列、审批动作、审核状态归属）。
- `version` 是内容载体（正文快照、可发布内容、版本关系归属）。
- 禁止将审核状态直接挂在 `version` 替代 `submission` 审核流。

## DCP-003 集成测试 DB 运行方式

- 集成测试数据库固定为 Docker Postgres 测试实例。
- 不使用内存 DB、Mock DB、宿主机共享开发库作为集成测试主路径。
- CI 与本地使用同一启动约定（同镜像、同初始化流程、独立测试库名）。

## DCP-004 `packages/db` 最小依赖与脚本清单

最小依赖（必须）：

- `drizzle-orm`
- `drizzle-kit`
- `pg`

脚本约定（至少）：

- `db:generate`：基于 schema 生成迁移。
- `db:migrate`：对目标数据库执行迁移。
- `db:push`：开发态直推 schema（仅开发环境）。
- `db:studio`：本地打开 drizzle studio（可选但建议保留）。
- `db:test:up`：启动 Docker Postgres 测试实例。
- `db:test:migrate`：对测试库执行迁移。
- `db:test`：运行集成测试（依赖测试库已就绪）。
- `db:test:down`：停止并清理测试实例。

## 对 Task 5 的直接影响

- Task 5 的数据模型必须包含 `submissions`，并建立与 `versions` 的明确关联字段。
- Task 5 审核接口命名与状态流转必须围绕 `submission`，禁止以 `version` 直接承载审批动作。
- Task 5 集成测试必须以 Docker Postgres 为执行基座，迁移与测试命令纳入同一流水线。
- Task 5 在 `packages/db` 初始化时即落齐最小依赖与脚本，避免后续补脚本导致任务拆返。
