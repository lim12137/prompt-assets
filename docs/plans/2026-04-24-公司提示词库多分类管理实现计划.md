1. 页面名称统一改为“公司提示词库”。
2. 分类模型升级为多对多（提示词支持多选分类标签）。
3. 管理端支持分类新增/删除。

5. 删除分类后零分类提示词自动归入系统“待分类（uncategorized）”。
6. 前台分类筛选支持多选 OR。
7. “待分类”默认折叠且不可手动勾选。

## 3.1 分类管理/删除确认/筛选 API 契约表

| 场景 | 方法与路径 | 请求 | 成功响应（最小） | 错误码（最小） | 备注 |
| --- | --- | --- | --- | --- | --- |
| 分类列表 | `GET /api/admin/categories` | Header: `x-user-role=admin` | `200 { categories: [{ slug,name,isSystem,isSelectable,isCollapsedByDefault,promptCount }] }` | `403 admin_role_required` | 管理端分类管理页初始化数据源 |
| 新增分类 | `POST /api/admin/categories` | Body: `{ name, slug? }` | `201 { category: { slug,name,isSystem:false,isSelectable:true } }` | `400 invalid_request` `403 admin_role_required` `409 category_slug_conflict` | `slug` 为空时后端自动生成 |
| 删除预检查 | `DELETE /api/admin/categories/{slug}` | Body: `{ confirm:false }`（默认） | `200 { dryRun:true, slug, impactedPromptCount, willBeUncategorizedCount, confirmationToken, confirmationExpiresAt }` | `403 admin_role_required` `403 system_category_forbidden` `404 category_not_found` | 删除前必须先拿预检查结果 |
| 删除确认执行 | `DELETE /api/admin/categories/{slug}` | Body: `{ confirm:true, confirmationToken }` | `200 { deleted:true, slug, impactedPromptCount, autoAssignedUncategorizedCount }` | `400 category_delete_confirmation_required` `400 invalid_confirmation_token` `409 category_delete_conflict` | 确认 token 与预检查绑定 |
| 前台筛选 | `GET /api/prompts?categories=prog,design&keyword=...&sort=...` | Query: `categories`（逗号分隔或重复 key） | `200 [{ ..., categories:[{slug,name}] }]` | `400 invalid_categories_filter` | 分类命中语义为 OR；与 keyword 为 AND |

### 删除确认协议（确认协议）

1. 客户端先发起 `confirm:false`（或省略 confirm）获取 `impactedPromptCount/willBeUncategorizedCount` 与 `confirmationToken`。  
2. 用户在 UI 二次确认后，客户端携带 `confirm:true + confirmationToken` 再发起删除。  
3. 服务端校验 token 与 `slug + 影响统计快照 + 过期时间` 一致；不一致返回 `invalid_confirmation_token`。  
4. 执行删除必须在事务内完成，返回最终统计并落审计日志。  

## 3.2 迁移切换闸门（双写/双读、切换条件、回滚标准、旧字段退场）

### 阶段策略

- 阶段 A（Expand）  
  - 上线 `prompt_categories` 与系统分类字段。  
  - 执行历史回填：`prompts.category_id -> prompt_categories`。  
  - 写路径进入双写：新建/导入/编辑同时写旧字段（兼容）与关系表（主目标）。  
  - 读路径双读：优先关系表，缺失时回退旧字段。
- 阶段 B（Switch Read）  
  - 全量校验通过后，读路径切换为关系表单读。  
  - 保留旧字段写入 1 个观察窗口（建议 3~7 天）用于回滚。
- 阶段 C（Stop Legacy Write）  
  - 停止旧字段写入，仅保留关系表写入。
- 阶段 D（Retire Legacy）  
  - 删除 `prompts.category_id` 与相关兼容代码。

### 切换条件（Gate）

- `prompt_categories` 行数 >= 已发布提示词数（每条至少 1 分类）。  
- 抽样与全量核对：列表/详情分类信息一致率 100%。  
- 关键自动化（分类管理、创建导入、前台筛选）连续 N 次绿灯（建议 N>=3）。  
- 无零分类脏数据（排除系统兜底后仍为空的异常数据）。

### 回滚标准

- 切换后出现任一阻断：分类丢失、筛选结果明显偏差、删除兜底失败、P1/P0 报错。  
- 回滚动作：立即将读路径切回双读/旧字段优先；恢复旧字段写入；冻结退场任务。  
- 回滚后需补“差异数据修复脚本 + 再验证报告”后才能再次切换。

### 旧字段退场标准

- 关系表单读已稳定通过一个完整发布周期。  
- 无消费者再依赖 `prompts.category_id`（API/脚本/报表全部确认）。  
- QA 与生产观测均无分类缺失告警。  
- 已完成一次可演练回滚预案并验证可执行。

## 3.3 DTO 口径与兼容期处理

### 目标 DTO（统一口径）

- 列表项：`categories: Array<{ slug: string; name: string }>`  
- 详情页：`categories: Array<{ slug: string; name: string }>`  
- 创建接口入参：优先 `categorySlugs: string[]`  
- 导入接口入参：优先 `categorySlugs: string[]`

### 兼容期策略

- 兼容读取：保留 `categorySlug/categoryName`（从 `categories[0]` 映射，优先非系统分类）。  
- 兼容写入：允许旧单值 `categorySlug`，后端统一转换为 `categorySlugs=[categorySlug]`。  
- 兼容冲突规则：若同时传 `categorySlug` 与 `categorySlugs`，以 `categorySlugs` 为准。  
- 退场条件：所有调用方切换到 `categories[]/categorySlugs[]` 后移除旧字段。

## 3.4 删除统计定义（必须统一）

- `impactedPromptCount` 定义：  
  - 删除目标分类 `C` 前，存在关系 `(prompt_id, C)` 的 `prompt_id` 去重计数。  
- `willBeUncategorizedCount` 定义（预检查口径）：  
  - 在受影响提示词中，移除 `C` 后将不再拥有任何“可手动分类”关系的提示词数量。  
  - 该数量表示“本次删除将触发自动补挂 `uncategorized`”的提示词数（去重后）。  
- `autoAssignedUncategorizedCount` 定义（执行结果口径）：  
  - 删除事务中实际新增 `(prompt_id, uncategorized)` 关系的提示词数。  
- 约束：三个统计均以事务内快照计算并随响应返回，前后端不可各算各的。

## 3.5 OR 筛选 SQL/索引/去重/稳定排序约束

### 查询语义

- 分类多选为 OR：命中任一分类即可。  
- 与关键词/状态组合时采用 AND：`(分类 OR 条件) AND (关键词条件) AND (状态条件)`。

### SQL 约束（示例形态）

```sql
WITH matched_prompt_ids AS (
  SELECT DISTINCT pc.prompt_id
  FROM prompt_categories pc
  INNER JOIN categories c ON c.id = pc.category_id
  WHERE c.slug = ANY($1::text[])
)
SELECT p.id, p.slug, p.title, p.summary, p.likes_count, p.updated_at
FROM prompts p
WHERE p.status = 'published'
  AND ($1 IS NULL OR p.id IN (SELECT prompt_id FROM matched_prompt_ids))
  AND ($2 IS NULL OR (p.title ILIKE $2 OR p.summary ILIKE $2))
ORDER BY p.updated_at DESC, p.id DESC;
```

### 索引与去重要求

- `prompt_categories(category_id, prompt_id)`：服务 OR 分类过滤。  
- `prompt_categories(prompt_id, category_id)` 唯一索引：防重。  
- `categories(slug)` 唯一索引：slug 查找。  
- `prompts(status, updated_at DESC, id DESC)`：稳定分页排序。  
- 若有热门排序，补 `prompts(status, likes_count DESC, updated_at DESC, id DESC)`。

### 稳定排序要求

- 最新排序：`updated_at DESC, id DESC`。  
- 热门排序：`likes_count DESC, updated_at DESC, id DESC`。  
- 任何 OR join 产生的重复行必须在 SQL 层去重（`DISTINCT`/`GROUP BY`/先取 ID 再回表）。

- `uncategorized` 规则需后端强约束，前端禁用仅做体验增强。
