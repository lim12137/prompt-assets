# MVP需求与信息架构

## 1. 设计基线

本文件用于指导后续产品设计、前端实现、接口定义与测试用例编写。界面基线以已确认的 Cherry 风格截图为准：

- 左侧分类导航。
- 顶部搜索。
- 卡片网格。
- 深色后台。
- 右上角 `导入` / `管理` / `创建` 入口。

任何偏离以上基线的设计调整，都需要作为变更单独确认。

## 2. 信息架构总览

建议采用以下一级页面结构：

1. 首页 / 提示词列表页
2. 提示词详情页
3. 提交改进版页面或弹窗
4. 管理视图
5. 导入页面或导入弹窗

推荐导航关系：

```text
首页（分类 + 搜索 + 卡片网格）
├─ 点击卡片 -> 提示词详情
│  ├─ 当前版本内容
│  ├─ 版本历史
│  └─ 提交改进版
├─ 右上角 创建 -> 新建提示词
├─ 右上角 导入 -> 导入提示词/版本
└─ 右上角 管理 -> 管理视图
```

## 3. 核心对象定义

### 3.1 分类 Category

字段建议：

- `id`
- `name`
- `slug`
- `sort_order`
- `description`
- `status`

约束：

- MVP 至少支持一级分类。
- 是否支持分类图标、颜色、二级分类，`待确认`。

### 3.2 提示词 Prompt

字段建议：

- `id`
- `slug`
- `title`
- `summary`
- `category_id`
- `current_version_id`
- `rating`
- `likes`
- `status`
- `created_at`
- `updated_at`

说明：

- `summary` 用于卡片展示，避免列表直接渲染长文本。
- `current_version_id` 指向当前生效版本。
- `rating` 与 `likes` 是否同时保留，当前建议都保留；产品口径 `待确认`。

### 3.3 版本 PromptVersion

字段建议：

- `id`
- `prompt_id`
- `version_no`
- `content`
- `change_note`
- `submitted_by`
- `submitted_at`
- `review_status`
- `reviewed_by`
- `reviewed_at`
- `meta`

`review_status` 建议枚举：

- `pending`
- `approved`
- `rejected`

### 3.4 用户反馈 Feedback

MVP 仅保留最小能力：

- 点赞或评分提交记录。
- 是否允许重复点赞，`待确认`。
- 是否记录评分历史，`待确认`。

## 4. 页面级需求

## 4.1 首页 / 提示词列表页

### 页面目标

- 让用户快速浏览、筛选、搜索和进入提示词详情。

### 布局要求

- 左侧：分类导航，支持分类切换和当前分类高亮。
- 顶部：全局搜索框，右上角展示 `导入` / `管理` / `创建`。
- 中间：卡片网格列表。
- 整体：深色背景，卡片与背景层级清晰。

### 卡片字段

每张卡片至少展示：

- 标题
- 分类名
- 简述
- 点赞数或评分
- 最近更新时间
- 复制按钮
- 进入详情的点击区域

### 交互要求

- 点击分类后刷新卡片列表。
- 输入搜索词后按标题、摘要、正文进行匹配。
- 点击复制后给出明确反馈。
- 点击卡片进入详情页。

### 空状态

- 当前分类下无数据时，显示空状态说明。
- 搜索无结果时，提供清空搜索入口。

## 4.2 提示词详情页

### 页面目标

- 展示当前生效版本内容。
- 提供复制、评分/点赞、查看版本历史、提交改进版入口。

### 页面模块

- 基本信息区：标题、分类、评分/点赞、最近更新时间。
- 当前版本区：完整提示词正文。
- 操作区：复制、点赞/评分、提交改进版。
- 版本历史区：版本列表 + 版本详情。

### 版本历史交互

- 默认按时间倒序显示。
- 点击某个版本后，右侧展示内容、变更说明、提交人、提交时间、审核状态。
- 当前版本需有明显标识。

## 4.3 提交改进版

### 页面目标

- 允许普通用户基于当前版本提交新版本候选。

### 表单字段

- 原版本号，只读。
- 新内容。
- 变更说明。
- 提交人。

### 提交结果

- 普通用户提交后写入 `pending` 状态。
- 管理员提交时是否可直接发布，`待确认`。

## 4.4 创建提示词

### 页面目标

- 允许管理员从零创建新的提示词条目。

### 表单字段

- 标题
- 分类
- 简述
- 初始内容
- 初始版本说明

### 成功条件

- 创建后自动生成第一个版本。
- 新条目可在首页卡片列表中立即可见，是否默认可见 `待确认`。

## 4.5 导入

### 页面目标

- 批量导入提示词或其版本数据，减少手工录入。

### MVP 约束

- 仅要求预留入口与最小流程。
- 导入格式、字段映射、失败回滚策略均为 `待确认`。

## 4.6 管理视图

### 页面目标

- 供管理员查看待审核版本、分类维护、提示词维护。

### MVP 最小模块

- 待审核版本列表。
- 分类管理。
- 提示词列表管理。

### 可延后模块

- 操作日志。
- 数据统计看板。
- 批量审核。

## 5. 关键业务流程

## 5.1 浏览与复制

1. 用户进入首页。
2. 在左侧选择分类或在顶部输入关键词。
3. 系统返回匹配卡片。
4. 用户点击卡片进入详情页。
5. 用户点击复制，系统返回成功提示。

## 5.2 提交改进版

1. 用户在详情页点击“提交改进版”。
2. 系统以当前版本内容作为初始值打开表单。
3. 用户修改内容并填写变更说明。
4. 系统保存为新版本候选，状态为 `pending`。
5. 管理员审核通过后，该版本成为可见版本历史的一部分。

## 5.3 审核发布

1. 管理员进入管理视图。
2. 打开待审核版本列表。
3. 查看内容、变更说明、提交人。
4. 执行通过或拒绝。
5. 通过后更新当前生效版本；是否自动替换当前版本，`待确认`。

## 6. 非功能要求

### 6.1 易用性

- 常用操作不超过 3 次点击。
- 搜索与复制应有即时反馈。
- 页面命名和按钮文案以中文为主。

### 6.2 部署

- 必须支持 Docker 化部署。
- 默认场景为内网部署。

### 6.3 可维护性

- 版本、分类、提示词对象需有明确字段边界。
- 后续新增标签、权限、导入格式时不应破坏现有结构。

## 7. 开发交付切分建议

建议按以下顺序实现：

1. 列表页与详情页静态结构。
2. 分类、搜索、卡片列表数据联动。
3. 复制与评分/点赞。
4. 版本历史读取。
5. 提交改进版。
6. 管理审核。
7. 导入能力。

## 8. 待确认项

- 搜索是否支持模糊匹配、拼音、标签、全文版本搜索。
- 点赞与评分的产品定义。
- 审核通过后是否自动切换当前版本。
- 导入支持的文件格式和校验机制。
- 管理端是否和前台共用布局。
- 是否需要保留被拒绝版本的前台可见性。
