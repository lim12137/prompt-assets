# 测试与验收方案

## 1. 目的

本文件用于约束后续实现阶段的测试方式、验收口径和测试报告落盘要求，确保 MVP 交付时可以按统一标准验证：

- 界面是否符合 Cherry 风格基线。
- 核心流程是否可用。
- 并发场景是否有明确测试与记录。
- 验收结论是否可追溯。

## 2. 测试范围

本期 MVP 至少覆盖以下范围：

- 左侧分类导航。
- 顶部搜索。
- 卡片网格展示。
- 深色后台视觉基线。
- 右上角 `导入` / `管理` / `创建` 入口。
- 提示词详情查看。
- 一键复制。
- 点赞/评分。
- 版本历史查看。
- 用户提交改进版。
- 管理员审核发布。
- Docker 化启动。

以下范围暂不作为必测项，除非后续需求确认进入本期：

- 企业 SSO / LDAP。
- 跨组织权限隔离。
- 高可用容灾。
- 移动端专用适配。

## 3. 测试类型

## 3.1 功能测试

验证每个页面和核心流程是否符合需求，包括：

- 分类切换是否正确刷新列表。
- 搜索是否返回正确结果。
- 复制按钮是否复制当前版本内容。
- 版本历史是否展示版本号、内容、说明、提交人、时间、审核状态。
- 提交改进版后是否生成 `pending` 状态版本。
- 管理员审核通过后是否正确生效。

## 3.2 界面与交互测试

重点验证是否符合已确认的界面基线：

- 左侧是否存在分类导航，且视觉上为常驻导航。
- 顶部是否存在全局搜索框。
- 主区是否为卡片网格，而非纯表格。
- 页面整体是否为深色后台风格。
- 右上角是否存在 `导入` / `管理` / `创建` 入口。

若与以上基线不一致，直接判定为不通过。

## 3.3 集成测试

验证前后端联动与状态流转：

- 首页列表与详情页数据一致。
- 当前版本与版本历史数据一致。
- 审核通过后首页展示内容同步更新。
- Docker 启动后页面、接口、数据存储路径可正常工作。

## 3.4 并发测试

并发测试为必需项，至少覆盖以下场景：

- 多用户同时搜索与浏览列表。
- 多用户同时复制同一提示词。
- 多用户同时提交同一提示词的新版本。
- 管理员审核与普通用户继续提交发生重叠。

并发测试目标：

- 不出现服务崩溃。
- 不出现版本号重复。
- 不出现当前版本指向错误。
- 不出现已审核结果被无意覆盖。

## 4. 测试环境要求

最低要求：

- 使用 Docker 或 Docker Compose 启动完整环境。
- 指定测试数据集，至少包含 3 个分类、10 条提示词、每条 2 个以上历史版本。
- 测试环境需与内网部署方式一致或高度接近。

待确认项：

- 是否需要单独的预发布环境，`待确认`。
- 是否需要真实身份系统联调环境，`待确认`。

## 5. 测试数据准备

建议最小测试数据集：

- 分类：内容创作、编程、设计。
- 提示词总数：不少于 10 条。
- 每个分类至少 3 条提示词。
- 至少 3 条提示词带有 2 个以上历史版本。
- 至少 2 条提示词存在 `pending` 状态版本。

## 6. 核心验收用例

## 6.1 首页结构验收

验收步骤：

1. 打开首页。
2. 检查左侧分类导航是否展示。
3. 检查顶部搜索框是否展示。
4. 检查内容区是否为卡片网格。
5. 检查整体是否为深色后台。
6. 检查右上角是否存在 `导入` / `管理` / `创建`。

通过标准：

- 6 项全部满足。

## 6.2 搜索与分类联动验收

验收步骤：

1. 点击一个分类。
2. 记录卡片列表结果。
3. 输入关键词搜索。
4. 验证结果与分类、关键词匹配。

通过标准：

- 分类和搜索条件共同生效。
- 无匹配结果时有明确空状态提示。

## 6.3 复制验收

验收步骤：

1. 打开任一提示词详情页。
2. 点击复制按钮。
3. 将剪贴板内容粘贴到验证区域。

通过标准：

- 剪贴板内容等于当前生效版本内容。

## 6.4 版本历史验收

验收步骤：

1. 打开带多版本的提示词详情页。
2. 进入版本历史区域。
3. 切换不同版本。

通过标准：

- 每个版本均可查看内容、变更说明、提交人、提交时间、审核状态。
- 当前版本有明确标识。

## 6.5 提交改进版与审核验收

验收步骤：

1. 普通用户提交一个改进版。
2. 检查系统生成 `pending` 版本。
3. 管理员在管理视图审核通过。
4. 返回详情页确认版本历史更新。

通过标准：

- 提交成功。
- 新版本状态正确。
- 审核结果可见。
- 当前生效版本更新逻辑符合产品设定；若未定，标记为 `待确认` 后不判通过。

## 6.6 Docker 启动验收

验收步骤：

1. 执行约定启动命令。
2. 检查容器是否全部启动。
3. 访问页面与关键接口。

通过标准：

- 启动命令可重复执行。
- 页面可访问。
- 核心流程无阻塞错误。

## 7. 并发测试报告落盘规范

本项目明确要求：并发测试必须形成 Markdown 报告，并落盘到 `docs/*.md`。

### 7.1 报告存放规则

- 存放目录：`D:\1work\提示词管理\docs`
- 文件命名建议：`并发测试报告-YYYYMMDD-场景名.md`
- 一个场景一份报告，或一次完整测试一份总报告，二选一。

### 7.2 报告必填内容

每份并发测试报告必须包含：

- 测试日期。
- 测试环境说明。
- 测试对象与场景。
- 测试命令。
- 并发参数。
- 结果摘要。
- 失败明细或异常现象。
- 最终结论。

### 7.3 报告模板要求

建议使用以下结构：

```md
# 并发测试报告-搜索与提交版本

## 测试日期

2026-04-23

## 测试环境

- Docker Compose
- 测试数据集版本：v1

## 测试场景

- 20 个并发用户同时搜索
- 10 个并发用户同时提交同一提示词的新版本

## 测试命令

```bash
# 在此填写实际执行命令
```

## 结果摘要

- 搜索接口成功率 100%
- 提交版本接口成功率 100%
- 未出现重复版本号

## 异常与问题

- 无

## 结论

- 通过
```

### 7.4 命令与结果摘要要求

报告中必须写明实际执行的测试命令和结果摘要，不允许只写“已测试通过”。

最低要求：

- 命令可复现。
- 并发数可识别。
- 结果摘要至少包含成功率、主要错误数、是否影响验收。

## 8. 验收输出物

验收完成时，至少应提交以下输出物：

- 功能测试结论。
- 界面基线检查结论。
- 集成测试结论。
- 并发测试 Markdown 报告，存放于 `docs/*.md`。
- 最终验收结论。

## 9. 验收判定规则

满足以下条件才可判定 MVP 验收通过：

- 核心功能测试通过。
- 界面基线检查通过。
- Docker 启动通过。
- 并发测试已执行且报告已落盘到 `docs/*.md`。
- 所有 `待确认` 项均未影响当前 MVP 主流程。

以下情况直接判定不通过：

- 页面结构偏离已确认界面基线。
- 复制、版本历史、提交改进版、审核流程任一主链路不可用。
- 并发测试未执行，或未形成落盘报告。

## 10. 待确认项

- 并发测试工具选型。
- 审核通过后当前版本切换策略。
- 点赞与评分的数据一致性规则。
- 导入流程是否纳入本期正式验收。
